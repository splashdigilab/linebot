# 待辦清單（依優先順序）

> 最後更新：2026-07-15（已完成：組織總覽、發票搬組織層、死路頁改造、OA 數量上限）
> 排序原則：**先排「有前置時間、你不動它就一直卡著」的事**，再排「不做會扣錯錢」的，
> 再排「服務已付費客戶」的，最後才是成長與商業完整性。

---

## 🚨 第 0 順位 — 今天就送出（不是工程工作，但不送就全卡住）

藍新與 ezPay 都是**申請制**，送件到開通有數天到數週的前置時間。這幾件事本身花不到一小時，
但你晚一天送，整個收款上線就晚一天。**先送件，再回來寫程式。**

1. [ ] **藍新：申請開通「定期定額支付工具」**
2. [ ] **藍新：申請開通「修改委託狀態 API」（AlterStatus / NPA-B051）**
       ⚠️ 這支比定期定額本身更不能少。沒有它，**客戶無法自助取消訂閱**，而且我們也無法在降級時
       終止委託 → 會出現「服務被降級、信用卡卻繼續被扣」，這是會上新聞的那種錯。
3. [ ] **申請 ezPay 電子發票商店帳號**（與金流是**不同**的一組金鑰，要另外申請）
4. [ ] **問會計：方案價是含稅還是未稅？**
       `plans.ts` 原本註解寫「未稅」，但結帳直接刷 `priceMonthly`。我按**含稅**實作
       （499 = 銷售額 475 + 稅額 24）。搞錯是稅務問題，不是體驗問題。

## 🔴 第 1 順位 — 不做就會扣錯錢

5. [ ] **拿藍新測試特店實機跑通一輪**
       P0/P1 的程式碼**一行都沒有對真實金流跑過**。金流的坑幾乎都在位元組層級——補位方式、
       欄位名大小寫、charset、時間戳 120 秒容差——單元測試驗不到。
       至少要跑通：建立委託 → 首期扣款 → 收到 Notify → 方案開通 → 開出發票 → 取消訂閱。
6. [ ] **設定環境變數**：`NEWEBPAY_PERIOD_ENABLED=true`、`NEWEBPAY_PERIOD_API_URL`、
       `NEWEBPAY_PERIOD_ALTER_URL`、`EZPAY_INVOICE_MERCHANT_ID` / `_HASH_KEY` / `_HASH_IV` / `_API_URL`
       ⚠️ `EZPAY_INVOICE_API_URL` 刻意沒有預設值——有預設值的話，正式環境漏設就會把**真實發票開到測試平台**。
7. [ ] **檢查有沒有「沒掛訂閱」的 workspace**
       拆掉老戶保護後門之後，這種帳號會從「吃到飽」變成「被攔在 200 則」。自己的測試 / demo OA
       若中招，去 super admin 後台指派 `test` 或 `internal` 方案（那兩個是真無限）。

## 🟠 第 2 順位 — 最會帶營收的那一個（很便宜，卻一直沒做）

8. [ ] **額度快用完時的升級提示**
       `derivePlanState` 早就有 `QUOTA_NEAR_THRESHOLD = 80` 和 `near` 狀態，但**沒有任何地方
       拿它去推升級**。免費 200 則用到 80% 的那一刻，是整個產品裡轉換率最高的時機——客人正在用、
       正覺得有用、正要不夠用。現在那一刻什麼都沒發生。
       做法：對話頁 / 用量頁頂端出現「本期額度剩 40 則，升級繼續」+ 直接開方案對話框。

## 🟡 第 3 順位 — 服務「已經付錢的人」

已付費客戶的體驗，優先於還沒進來的客戶。

9. [x] ✅ **組織總覽頁**（給 org admin，不是 super admin）
       org admin 在後端有實權（是組織下所有 OA 的 admin、可建新 OA），但 UI 只有工作區選擇頁上
       那一個「+」按鈕。代理商管 10 個 OA，現在要一個一個點進去看，**沒有任何總覽**。
       內容：每個 OA 一張卡 — 方案 / 本期用量 / 額度警示 / LINE 接上了沒。
10. [x] ✅ **發票統編搬到組織層**（設計債）
       我在 P1 把發票資訊做在 workspace 層。但**統編幾乎一定是組織層級的**——一家公司開 3 個 OA，
       不會想填 3 次統編。改成「組織預設 + OA 可覆寫」。
11. [ ] **組織帳務彙總**：所有 OA 的下次扣款、付款紀錄、發票號碼
12. [ ] **組織成員管理**：讓客戶自己加組織管理員（現在**只有 super admin 能加**，客戶每次要加同事都得來找你）

## 🟢 第 4 順位 — 止住白白流失的訪客

13. [x] ✅ **把登入後的死路變成入口**
        現在沒權限的人登入後看到「你目前沒有任何官方帳號的存取權限。請聯繫管理員邀請你加入。」
        ——這是寫給系統看的錯誤訊息。而你的登入按鈕就放在首頁上，**是你們把他請進來，然後給他一堵牆。**
        改成：承認目前是邀請制、**顯示他登入用的信箱（可一鍵複製）**（同事要邀請他就是需要這個）、
        給一個真出口（預約 Demo / 聯繫我們）。
        ⚠️ **不要在這裡放購買按鈕**——他還沒有 OA、沒接 LINE、沒看過機器人回話，等於叫還沒有車的人先買油。
14. [x] ✅ 登入頁補一句「目前為邀請制」，別讓人走進一個他不該走進的門

## ⚪ 第 7 順位 — 把商業模式補完整

24. [ ] **超量加購：讓它真的能收錢，或從方案表上拿掉**
        `plans.ts` 每個付費方案都寫著 `overagePerReply: 0.8`，前端也顯示「超量加購 NT$0.8/則」，
        但程式的實際行為是**則數用完一律轉真人**——既不讓他繼續用，也不跟他收錢。
        這是張開了卻兌現不了的支票，而且撞頂的正是你最賺錢的客戶。
25. [ ] **換方案的按比例補差額（proration）**
        目前換方案是「重新起算一整期，舊方案剩餘天數作廢」。升級的人拿到完整一期不吃虧，所以現在安全。
26. [ ] **期中降級** — ⚠️ **必須先做 25**，否則客戶會損失已付的剩餘天數
27. [ ] **年繳方案**（8 折換現金流與留存）— `plans.ts` 目前只有 `priceMonthly`
28. [ ] **自助試用流程** — `trialing` 狀態早就存在，但沒有任何流程用到它

## 🔧 技術債（不急，但別忘了）

29. [ ] **`OrgMemberDoc` 的型別在說謊**
        `shared/types/organization.ts:58-72` 宣告 doc ID 是 `${uid}_${orgId}` 且有必填的 `uid` 欄位，
        但兩個寫入端都是用 `.add()` 自動 ID、而且**根本沒寫 `uid`**。讀取走 email，所以功能是好的，
        但型別是錯的——下一個人照著型別寫就會踩坑。
30. [ ] **`owner` 角色沒有任何專屬權限**
        `shared/permissions.ts` 的 `CAPABILITIES` 裡沒有一項要求 `owner`——它只是等級 4，
        能做的事跟 `admin` 一模一樣。要嘛給它專屬權限（刪除 OA、轉移擁有權），要嘛承認它只是個標籤。
